## 3. Performance (62/100)

### 3.1 Base de données (58/100)

**Pool :** `connectionLimit=10`, singleton global via `globalThis` (correct).
**Pas de N+1** : JOINs utilisés correctement.

**Problèmes :**
- `facturation.ts:40-71` : COUNT + SELECT avec 5 JOINs → lent sur gros volumes
- `bdc.ts:34-49` : 2 requêtes séparées (COUNT + SELECT) → 2 round-trips
- Pagination LIMIT/OFFSET → dégradation sur OFFSET élevé (page 100+)
- Index à vérifier : `J2_date_CMES`, `gdc_itemStatus`, `gdc_hidden`, `date_factu_anticipee`

### 3.2 React Rendering (55/100)

**Problèmes majeurs :**

| Fichier | Problème | Impact |
|---------|----------|--------|
| `FacturationTable.tsx:229` | `columns` recréé à chaque render (array inline) | Re-render de toutes les cellules |
| `BdcTable.tsx:16-48` | Idem | Idem |
| `ResourcesTable.tsx:24-76` | Idem | Idem |
| `DataTable.tsx:105-119` | Map sans virtualisation → 50+ lignes = 50+ TR DOM | Coût DOM élevé |
| `ContractPanel.tsx:91-94` | Re-fetch à chaque changement de search | Requêtes redondantes |

### 3.3 Data Fetching / SWR (65/100)

- **Pas de `SWRConfig` global** : pas de `dedupInterval`, `revalidateOnFocus`, `errorRetryCount`
- `refreshInterval: 30000` même si tab inactive → requêtes inutiles
- `useCancelledBdcIds` : `new Set()` recréé à chaque render → re-render en cascade
- `SiteSearchModal.tsx` : fetch manuel au lieu de SWR → pas de cache ni déduplication
- Pas de `keepPreviousData` → flash "loading" à chaque nouvelle recherche

### 3.4 API Sophia (70/100)

- Pas de cache côté serveur (ni Redis ni LRU)
- Risque de refresh concurrent du token (pas de mutex/lock)
- Token bien géré sinon (refresh 1 min avant expiry)
